# 作业/小型项目 AI 开发流程指南

> 适用场景：个人作业、课程项目、小工具、原型验证
> 核心思想：省掉不必要的文档，快速出代码，但关键步骤不缺

---

## 大型 vs 小型 对比

| 维度 | 大型全栈项目 | 作业/小型项目 |
|------|-------------|--------------|
| 周期 | >2 周 | 1-3 天 |
| 复杂度 | 多模块、多角色 | 单模块、单用户 |
| 文档 | 每步有产出物 | 只需关键文档 |
| 流程 | 八步 | 三步（合并大部分步骤） |
| 审查 | 多轮审查 | 写完即查 |
| 部署 | Docker + CI/CD | 能跑就行 |

---

## 小型项目三步流程

```
需求澄清 ──→ 直接开发 ──→ 交付
  │              │            │
  │              │            │
/bdd      写代码+测试+审查   /quick-package
         （合并步骤2-7）    部署或打包
```

---

## 第一步：需求澄清（15-30分钟）

**和大型项目一样，不能省略。**

### 做什么

把作业要求变成清晰的功能清单。

### 手动调用

| 情况 | 用什么 |
|------|--------|
| 作业要求明确 | 直接写需求文档 |
| 选题不明确 | `/brainstorming` 探索 |
| 需要梳理功能 | `/bdd` |

### 产出物

**不需要 .internal/ 目录，直接在对话中确认：**

```markdown
## 作业需求

### 功能清单
- [ ] 功能A
- [ ] 功能B
- [ ] 功能C

### 技术栈
React + Vite + TypeScript（如果老师指定）
Python + Flask（如果自选）
```

### 检查清单

- [ ] 功能清单列出来了
- [ ] 技术栈确定好了
- [ ] 验收标准清楚了（怎么算及格）

---

## 第二步：直接开发（主要时间）

**大型项目的第2-7步合并为这一步。**

### 开发流程

```
1. 说一句"开始开发"，AI 直接初始化项目
2. 逐个功能模块开发
3. 每个模块：写代码 → 测试 → 审查
4. 全做完后跑一遍验收测试
```

### 手动调用

| 阶段 | 用什么 | 什么时候用 |
|------|--------|-----------|
| 初始化项目 | 直接说"创建 React 项目" | 开始时 |
| 开发功能 | 直接说"实现 XX 功能" | 每个功能 |
| 写测试 | `/test-driven-development` | 需要测试时（考试要求） |
| 调试 bug | `/systematic-debugging` | 遇到错误时 |
| 前端问题 | `/frontend-debug` | 前端显示异常 |
| 代码审查 | `/code-audit` | 全部功能完成后 |
| 简单审查 | `/simplify` | 感觉代码冗余时 |
| 上下文管理 | `/context-compact` | 工具调用 > 40 次 |

### 注意事项

**1. 不需要的步骤**
- 技术选型 → 直接用最熟悉的框架
- 架构设计 → 小型项目不需要复杂架构
- 骨架搭建 → 直接用脚手架
- 多 Agent 并行 → 一个人就够了

**2. 必须保留的**
- 写完代码后审查：`/code-audit`
- 遇到 bug 时系统调试：`/systematic-debugging`
- 上下文快满时压缩：`/context-compact`

**3. 测试可简化**
- 如果作业不要求写测试，可以只做手工验证
- 如果要求写测试，用 `/test-driven-development`
- 不需要 80% 覆盖率，只要覆盖核心逻辑

**4. 提交节奏**
- 完成每个功能后提交一次
- 不要在最后一次性提交

---

## 第三步：交付（30分钟）

### 做什么

打包、验证、提交。

### 手动调用

| 做什么 | 用什么 |
|--------|--------|
| 打压缩包 | `/quick-package` |
| 部署到 Vercel | `/vercel-deploy` |
| 写 README | 直接让 AI 生成 |

### 交付清单

```markdown
- [ ] 功能全部跑通（手动测试过）
- [ ] README 写好了（怎么运行、功能说明）
- [ ] 打好了压缩包（如果需要交）
- [ ] 代码已提交（git commit）
- [ ] 作业要求全部满足
```

## 不同作业类型的处理

### 类型一：前端页面作业

```
第一步：确认页面内容和交互
第二步：/frontend-design 直接创建页面
第三步：/quick-package 打包
```

**典型流程**：
```
1. "帮我创建一个登录注册页面，用 React + Ant Design"
2. AI 直接写代码
3. 看效果，改细节
4. /quick-package 打包上交
```

### 类型二：后端 API 作业

```
第一步：确认接口列表
第二步：直接写 API 代码
第三步：手动测试 + 打包
```

**典型流程**：
```
1. "帮我写一个学生管理系统 API，用 FastAPI"
2. AI 直接写 CRUD 接口
3. 用 curl 测试每个接口
4. /quick-package 打包上交
```

### 类型三：全栈小项目

```
第一步：确认功能和技术栈
第二步：先后端再前端，或并行
第三步：联调 + 打包
```

**典型流程**：
```
1. "帮我做一个简单的博客系统，React + FastAPI"
2. 先开发后端 API
3. 再开发前端页面
4. 联调验证
5. /quick-package 打包上交
```

### 类型四：数据分析/算法作业

```
第一步：确认数据和分析要求
第二步：AI 直接写 Python 脚本
第三步：验证结果 + 打包
```

**典型流程**：
```
1. "帮我分析这个 CSV 文件，用 Python"
2. AI 写数据分析脚本
3. E:/conda_envs/cv/python.exe script.py 运行验证
4. /quick-package 打包上交
```

## 典型对话示范

### 场景：课程作业，做一个待办事项应用

```
Boss: 帮我做一个待办事项应用，要求有添加、完成、删除功能

AI: Boss，先确认一下需求：
1. 用 React 还是 Vue？
2. 需要后端吗？还是纯前端？
3. 数据存哪里？localStorage 还是数据库？
4. 有没有参考界面？

Boss: React，纯前端，存 localStorage。简单就行

AI: 明白了。功能清单确认：
- [ ] 添加待办
- [ ] 标记完成
- [ ] 删除待办
- [ ] 显示列表

开始开发。

[AI 直接写代码...]

Boss: 跑一下看看

[AI 启动项目，截图验证]

Boss: 可以，打包上交

[AI 运行 /quick-package]
```

---

## 核心原则

1. **需求澄清不能省**：无论项目多小，先确认做什么
2. **直接写代码**：不要花时间在设计文档上
3. **功能做完再审查**：不要边写边审
4. **上下文管理**：工具调用多时及时压缩
5. **先跑通再优化**：不要过早追求完美

## 已知 Dex3 DDS 合同

每只 Dex3-1 手有 7 个电机。DDS 的 `HandCmd_.motor_cmd` 与 `HandState_.motor_state` 顺序为：

```text
[thumb_0, thumb_1, thumb_2, middle_0, middle_1, index_0, index_1]
```

话题为：

```text
rt/dex3/left/cmd     发布 HandCmd_
rt/dex3/right/cmd    发布 HandCmd_
rt/dex3/left/state   订阅 HandState_
rt/dex3/right/state  订阅 HandState_
```

位置单位为 rad；`q/dq/tau/kp/kd` 的确切控制语义、允许模式和限值必须使用实机 SDK 的当前定义确认。不要直接照抄历史参数写入 `HandCmd_`。

## 28D 合同的阻断风险

当前 π0.5 数据和服务使用：

```text
state/action = 左臂 7 + 右臂 7 + 左手 7 + 右手 7 = 28D
```

已核对的事实：

- 前 14 维双臂顺序可与 G1 官方 SDK `motor_cmd[15:29]` 的双臂区间对应。
- 当前数据每只手原始数值列顺序为 `[index0, middle0, thumb0, index1, middle1, thumb1, thumb2]`。
- Dex3 DDS 每只手的官方顺序是 `[thumb0, thumb1, thumb2, middle0, middle1, index0, index1]`。
- 数值列转向 DDS 的重排索引应为 `[2, 5, 6, 1, 4, 0, 3]`；左右手分别执行。
- 尚无逐关节真机零位、正方向、运动范围和单位的证据。因此“28D 已和真机完全一致”是未证实结论。

这不是文档瑕疵，而是实机部署阻断项。必须新增唯一的 `dataset/policy <-> DDS` 显式 adapter，并保留输入、输出、方向和限位的可测试表，不能在控制脚本里散落重排逻辑。

## 到机后的只读验收顺序

1. 记录机器人型号、序列号、固件、SDK 版本、操作系统、网卡和 DDS domain 配置。
2. 编译并运行官方 G1 与 Dex3 的**只读状态订阅**示例；此阶段不发布任何动作。
3. 连续记录至少 60 秒的 G1 state 与两只 HandState，检查频率、NaN、错误码、温度与断流。
4. 将每个实机状态向量与 28D adapter 的候选映射并排打印，只验证形状和单位。
5. 在官方安全流程、低速和单关节条件下，逐轴确认零位、正方向、限位和控制模式；每次只验证一个关节。
6. 只有完整映射表签字确认后，才允许启用“只发送零增量/保持位姿”的写入测试。

## 禁止事项

- 不把 `rt/lowcmd` 或 Dex3 `*/cmd` 指向真机做首次调试。
- 不通过跨网络的 A6000 直接承担低层 DDS 实时控制。
- 不用仿真关节索引、URDF 顺序或数据集标签代替实机逐轴确认。
- 不在没有急停、看门狗、动作超时和限幅器的情况下让 π0.5 动作进入执行链路。

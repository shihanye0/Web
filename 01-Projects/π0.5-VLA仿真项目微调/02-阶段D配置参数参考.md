## 冻结、精度与显存

| 参数 | 阶段 D 值 | 含义 |
|---|---|---|
| `train_expert_only` | `true` | 冻结整个 PaliGemma，只训练 action expert 与动作/时间投影 |
| `freeze_vision_encoder` | `true` | 显式冻结视觉塔；在 expert-only 下是重复但清晰的约束 |
| `use_peft` | `false` | 没有 LoRA/adapter，训练的是完整 action expert |
| `dtype` | `bfloat16` | 大部分模型用 BF16；源码保留视觉路径等部分为 FP32 |
| `gradient_checkpointing` | `true` | 用重复计算换激活显存，降低训练峰值 |
| `compile_model` | `false` | 未启用 `torch.compile` |
| `batch_size` | `1` | 12GB GPU 的保守设置 |
| `num_workers` | `0` | 主进程取数，减少多进程不确定性 |

## 优化器与学习率

| 参数                           |                     阶段 D 值 | 含义                        |
| ---------------------------- | -------------------------: | ------------------------- |
| `use_policy_training_preset` |                    `false` | 不使用 π0.5 默认 AdamW preset  |
| `optimizer.type`             |                      `sgd` | 仅为低显存工程 smoke             |
| `optimizer.lr`               |                   `2.5e-5` | 初始/峰值学习率                  |
| `optimizer.momentum`         |                      `0.0` | 不保存 momentum 状态，显著省显存     |
| `optimizer.weight_decay`     |                     `0.01` | SGD 参数衰减                  |
| `optimizer.grad_clip_norm`   |                      `1.0` | optimizer step 前裁剪总体梯度范数  |
| `scheduler.type`             | `cosine_decay_with_warmup` | 按 step 调整学习率              |
| `num_warmup_steps`           |                `1000 -> 0` | 20 步短跑自动缩放后的实际 warmup 为 0 |
| `num_decay_steps`            |              `30000 -> 20` | 自动缩放到本次训练总步数              |
| `decay_lr`                   |                   `2.5e-6` | 最低学习率                     |

## 当前没有做的事情

- 没有训练视觉编码器或语言模型。
- 没有 LoRA、QLoRA 或 8-bit optimizer。
- 没有图像增强。
- 没有梯度累积，effective batch size 仍为 1。
- 没有训练/验证 episode 划分。
- 没有仿真闭环成功率评估。
- 没有真实 G1/Dex3 执行。

## LoRA 支持的当前边界

LeRobot 的 PI05 提供 PEFT 默认目标：action expert 各层 attention 的 `q_proj/v_proj`，以及 `action_in_proj/action_out_proj`。默认正则表达式仍包含 PI0 风格的 `action_time_mlp_*` 名称，而当前 PI05 实际层名是 `time_mlp_in/out`，因此做 LoRA 实验前要打印 trainable parameter names，不能只相信“PEFT 已启用”。

## 最小验收字段

每次实验至少记录：数据版本、基础模型、可训练参数数、optimizer、实际 LR 日程、steps、seed、峰值显存、checkpoint 哈希、重载结果和独立评估指标。模板见 [[04-后续练习方向与实验记录模板]]。

## 三套配置的对比口径

本节中的“基础模型”指本机的 `models/pi05_base`（`config.json`）， “原项目正式训练”指 2026-07-03 生成的 `pi05_final_scene_7demo_1000` checkpoint（第 1000 步）， “阶段 D”指 2026-07-12 的 `pi05_final_scene_7demo_smoke20_local_sgd` checkpoint（第 20 步）。三者不是三个不同的网络架构：后两者都从基础 π0.5 权重初始化，差别主要在训练数据、可训练参数和优化过程。

## 基础模型 → 原项目正式训练：做了哪些微调

### 模型层

- **输入/输出合同变成养老照护任务合同**：输入是两路 RGB（左/右高位相机）加 28 维 G1/Dex3 状态，输出是 28 维动作；网络内部仍将状态和动作补零到 `max_*_dim=32`。
- **动作专家适配机器人动作空间**：保留 PaliGemma（Gemma 2B）和 Gemma action expert（300M）架构，只让 action expert 相关参数参与更新。实际可训练部分包括 action expert、`action_in_proj`、`action_out_proj` 和时间 MLP；PaliGemma 语言/视觉主干不更新。
- **冻结视觉塔**：`freeze_vision_encoder=true`，避免在小规模仿真数据上更新视觉特征。
- **使用数据集统计量做归一化**：processor 使用该数据集的 state/action quantile statistics；`use_relative_actions=false`，因此 loss 使用绝对动作值，不会在运行时自动把标签转换成相对动作。
- **任务文本和多视角图像进入条件输入**：养老照护场景文本由数据集提供，图像分辨率在 processor 中变为 `224x224`。

这属于“**动作专家微调**”，不是全参数微调，也不是 LoRA/QLoRA。基础模型本身的 `config.json` 只描述网络和默认超参数，并不包含本项目的 7 个 demonstration、机器人动作统计量或已训练的任务 checkpoint。

### 训练层

- 使用 7-demo 数据集 `final_scene_vla_care_7demo_filtered_stride2`，正式训练 `steps=1000`，每 `250` 步保存一个 checkpoint。
- 启用 BF16、gradient checkpointing、`batch_size=1`、`num_workers=0`。
- 使用 π0.5 的默认 AdamW preset：`lr=2.5e-5`、`betas=(0.9,0.95)`、`weight_decay=0.01`、`grad_clip_norm=1.0`，配合 cosine decay with warmup。

因此，基础模型到原项目的主要新增信息是“这个特定机器人、相机、状态统计和养老照护动作分布”，而不是新增一种模型结构。

| 基础架构 | Gemma 2B + Gemma 300M action expert | 相同 |
| 输入 | 两路 RGB + 28D state | 相同 |

### 阶段 D 的实际改动

| 改动            | 原项目正式训练                            | 阶段 D                                  | 影响判断                                                    |
| ------------- | ---------------------------------- | ------------------------------------- | ------------------------------------------------------- |
| 优化器           | AdamW，带 `betas` 和 optimizer state  | SGD，`momentum=0`                      | **会改变参数更新轨迹和最终权重**；选择它是为绕过 12GB RTX 4070 的 AdamW 状态 OOM |
| preset        | `use_policy_training_preset=true`  | `false`，显式写出 SGD/scheduler            | 训练算法配置改变，且便于复现低显存实验                                     |
| 训练长度          | `1000` 步                           | `20` 步 smoke                          | **决定性差异**；20 步只能验证链路，不能和正式效果比较                          |
| 保存            | 每 `250` 步                          | 第 `20` 步保存                            | 只影响产物频率，不改变单步梯度                                         |
| 学习率日程         | cosine，正式 1000 步                   | 同一 scheduler 类型，但按 20 步总长缩放           | 短跑 warmup 实际为 0，衰减终点被压缩到第 20 步                          |
| 数据根目录         | 旧 `/data/zhangxu/...`              | 本机 `/home/user/zhangxu/...`           | **仅运行路径变化**，数据内容和 repo_id 相同                            |
| 基础模型路径        | 旧 `/data/zhangxu/models/pi05_base` | `/home/user/zhangxu/models/pi05_base` | 仅路径修复；权重应由哈希确认相同                                        |
| 输出目录/Hub repo | `..._7demo_1000`                   | `..._smoke20_local_sgd`               | 产物隔离和命名变化，不是网络结构变化                                      |
| W&B、评估        | W&B 关闭，`eval_freq=0`               | 相同                                    | 没有新增在线评估能力                                              |

### 阶段 D 的定位

阶段 D 是“**低显存、20 步、动作专家 SGD smoke**”。它验证了 forward、backward、optimizer step、checkpoint 保存、processor 重载和离线评估的链路；它不是正式模型训练，也不能据此判断养老照护任务成功率。阶段 D 的 MAE `0.159767` 与基础模型 `0.159774` 几乎相同，符合仅训练 20 步的预期，不能解读为显著能力提升。

## 基础模型 → 阶段 D：完整结果差异

这条链路可以概括为：

```text
pi05_base（预训练通用视觉-语言-动作模型）
  └─ 载入 7-demo G1/Dex3 数据、双相机、28D 状态和数据集统计量
     └─ 冻结 PaliGemma/视觉塔，仅更新 action expert 及其投影/时间 MLP
        └─ 阶段 D 使用 SGD、batch=1、20 步，生成 smoke checkpoint
```

| 维度 | 基础模型 | 阶段 D |
|---|---|---|
| 权重状态 | 官方/预训练权重 | 基础权重上执行 20 次 SGD 更新后的权重 |
| 任务数据 | 不包含本项目数据 | 7-demo 仿真养老照护数据 |
| 机器人合同 | 通用 π0.5 配置 | G1/Dex3 28D、两路相机 |
| 可训练范围 | 作为初始化基线 | 仅 action expert 相关参数（约 6.934 亿/41.434 亿） |
| 优化器 | 配置默认 AdamW 能力 | 实际采用无 momentum SGD |
| 训练结论 | 只代表预训练能力 | 只代表训练链路 smoke，不代表收敛 |

## 哪些变化是真正的“模型微调”

### 会改变模型输出的变化

1. action expert 及投影/时间 MLP 的权重更新。
2. 训练数据中的 28D 状态、28D 动作、双相机图像、任务文本和 dataset statistics。
3. 优化器从 AdamW 改成 SGD、学习率日程和训练步数变化。
4. 若未来改 `use_relative_actions`、动作列顺序、归一化统计或相机语义，也会改变动作学习目标，必须单独版本化并重新评估。

### 不改变模型语义、但影响能否运行的变化

- `/data/zhangxu` → `/home/user/zhangxu` 的路径替换。
- 离线 Hugging Face cache、tokenizer 本地路径和输出目录隔离。
- `batch_size=1`、`num_workers=0`、关闭 W&B、关闭环境评估。
- `save_freq`、日志频率以及时间戳目录命名。

这些工程改动不能被写成“模型能力提升”；它们的验收证据应是命令成功、显存峰值、checkpoint 可重载和评估脚本可运行。

## 最终结论与后续学习重点

目前三者的关系不是“改了三套模型”，而是：

1. **基础模型**提供 π0.5 的通用先验。
2. **原项目正式训练**用真实项目数据分布完成动作专家适配，训练预算为 1000 步，优化器为 AdamW。
3. **阶段 D**在相同任务合同和冻结策略下，为本机 12GB 显存把优化器换成无 momentum SGD，并缩短为 20 步，重点是跑通训练链路。

后续练习应按以下顺序进行：先固定数据、processor、28D 列顺序和评估脚本，只比较 AdamW/SGD；再把 20 步 smoke 扩展到 250/1000 步；最后才尝试 LoRA、解冻视觉塔或相对动作。每次只改变一个变量，并记录 [[04-后续练习方向与实验记录模板]] 中的完整字段。

# cloud.md — 知识库增量处理规则

> 本文件是知识库的"生长规则"。Claude 每次处理新内容时，严格遵循本文件流程。
> 核心原则：**只处理增量，不重复扫描；只关联已有，不凭空创造。**

---

## 处理触发条件

当以下任一情况发生时，启动增量处理：
1. `00-Inbox` 中出现新文件
2. Boss 手动说"处理这条笔记"或"整理 Inbox"
3. Boss 说"把这篇文章加入知识库"

---

## 增量处理流程

### Step 1：扫描新增内容

```
只读取 00-Inbox 中未处理的文件（无 processed: true 标记的）
```

### Step 2：内容分类

| 类型 | 判断依据 | 目标文件夹 |
|------|---------|-----------|
| 论文/研究 | 有作者、期刊、DOI | `02-Papers/` |
| 概念/知识 | 解释某个技术或理论 | `03-Knowledge/` |
| 代码/工具 | 涉及代码实现或工具使用 | `05-Code/` |
| 项目相关 | 与当前项目直接相关 | `01-Projects/` |
| 日常记录 | 随手笔记、想法 | `06-Daily/` |

### Step 3：提取核心概念

对每条新内容，提取：
- **核心概念**：文中提到的关键术语、理论、方法
- **核心观点**：作者的主要论点或结论
- **关联点**：与已有知识库笔记可能的关联

### Step 4：概念去重与关联

```
对每个提取出的概念：
1. 在 INDEX.md 中查找是否已存在
2. 已存在 → 在已有笔记的"反向引用"中追加新来源
3. 不存在 → 创建新概念笔记（使用 07-Templates/concept-note.md 模板）
```

### Step 5：创建/更新笔记

**新概念笔记格式**（严格遵循模板）：

```markdown
---
title: "概念名"
date: YYYY-MM-DD
tags: [concept, 相关领域]
status: seed
sources: ["来源文件名"]
---

# 概念名

## 定义
一句话说清楚是什么。

## 核心要点
- 要点1
- 要点2

## 与已有知识的关联
- [[已有概念A]] — 关系说明
- [[已有概念B]] — 关系说明

## 来源
> 从 [[来源笔记]] 中提取

## 待深入
- [ ] 需要进一步了解的点
```

### Step 6：更新全局索引

每次处理完，更新 `INDEX.md`：
- 新增概念追加到对应分类下
- 更新关联关系

### Step 7：标记已处理

在原始 Inbox 文件的 frontmatter 中添加：
```yaml
processed: true
processed_date: YYYY-MM-DD
```

---

## 质量检查清单

每条新笔记必须满足：
- [ ] 有完整的 frontmatter（title, date, tags, status）
- [ ] 定义能一句话说清楚
- [ ] 至少关联 1 个已有概念（除非是全新的领域）
- [ ] 来源可追溯
- [ ] 不与已有笔记内容重复

---

## 概念笔记生命周期

```
seed → growing → mature → evergreen
```

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| seed | 刚创建，内容单薄 | 初始创建 |
| growing | 有多条来源，内容充实 | 3+ 条来源关联 |
| mature | 概念完整，关联丰富 | 5+ 条关联 + 自身内容完整 |
| evergreen | 核心概念，很少变动 | Boss 手动标记 |

---

## 禁止事项

1. **不重复处理** — 已标记 `processed: true` 的文件不再处理
2. **不凭空创造** — 不编造概念间不存在的关联
3. **不过度拆分** — 一个概念笔记足够说明的，不要拆成多个
4. **不丢失来源** — 每条信息必须可追溯到原始文件
5. **不自动删除** — 永远不删除已有笔记，只追加和更新

---

## 快捷命令

Boss 可以用以下命令触发处理：

| 命令 | 效果 |
|------|------|
| "处理 Inbox" | 扫描 00-Inbox，执行完整流程 |
| "把这条加入知识库" | 处理当前讨论的内容 |
| "更新索引" | 只重建 INDEX.md |
| "整理概念 X" | 深入扩展某个概念笔记 |

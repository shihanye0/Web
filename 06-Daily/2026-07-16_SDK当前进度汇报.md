## 一、结论

SDK 基础闭环已完成，当前已经具备：

```text
计算机结构化输入
→ 本机 schema / 允许列表解析
→ 官方 unitree_sdk2
→ G1 官方服务接收并执行
→ RPC / 播放状态回读
→ 计算机结构化输出
```

现有闭环只覆盖两个受限意图：

```json
{"intent":"get_volume"}
{"intent":"speak","phrase_id":"connection_ok"}
```

这表示 SDK 通信、查询、非运动可见响应和回读均有实机证据；它不等同于通用自然语言理解、持续任务服务、运动控制或 Dex3 控制。

## 三、当前 SDK 代码与 owner

### 1. 状态读取 owner

- `src/g1_state_monitor.hpp/.cpp`：状态校验、快照、频率、断流和事件语义。
- `src/g1_state_monitor_main.cpp`：官方 DDS `ChannelSubscriber` adapter，仅负责接线与控制台 JSON 输出。

该链路的责任是“G1 实体状态 → 计算机可观察状态”，不发命令。

### 2. 信息理解与分派 owner

- `src/g1_intent.hpp/.cpp`：唯一拥有输入 schema、字段校验、允许列表和单次分派语义。
- `src/g1_intent_cli.cpp`：官方 `AudioClient` adapter，执行 `GetVolume` 或固定 `TtsMaker`，并输出 JSON。
- `src/g1_command_runtime.hpp/.cpp`：唯一拥有单进程请求缓存、结果重放、ID 冲突和成功 TTS 速率门禁；当前以 injected adapter 完成离线验证。
- `src/g1_runtime.hpp/.cpp`：拥有受限 JSONL 接线、播放周期状态和 SDK runtime 生命周期顺序。
- `src/g1_sdk_runtime_main.cpp`：官方 `AudioClient` 与 `rt/audio_msg` 的薄 adapter，仅暴露两条冻结命令。

输入上界为 256 字节。空输入、非法 JSON、重复键、未知字段、未知意图、未知短语及类型错误均在 SDK 初始化前拒绝。当前没有任意 TTS 文本入口。

运行时核心已经接到官方 SDK adapter 并通过实机验收：同一进程内重发同一请求时，第二次返回 `replayed=true`，没有第二次播放状态；查询、固定播报与并行实体状态均有本轮 G1 端证据。

### 3. 当前可运行命令

```bash
cmake -S . -B /tmp/g1-gate5f-build -DCMAKE_BUILD_TYPE=RelWithDebInfo
cmake --build /tmp/g1-gate5f-build -j2
ctest --test-dir /tmp/g1-gate5f-build --output-on-failure
```

这三条命令分别生成构建目录、编译当前全部 target、运行 4 个本地测试；验证运行时核心、播放状态机和生命周期，不发送 G1 请求。

```bash
printf '%s\n' \
  '{"request_id":"volume-001","command":"查询音量"}' \
  '{"request_id":"speak-001","command":"播报连接成功"}' \
  '{"request_id":"speak-001","command":"播报连接成功"}' \
  | timeout 20s /tmp/g1-gate5f-build/g1_command_runtime_cli
```

该命令在同一进程中验证查询、固定播报和同 ID 重放。`rpc_result=0` 表示 SDK 调用成功；首次播报应有 `playback_started=true` 与 `playback_stopped=true`；第三行应为 `replayed=true`，不产生第二次播放。

## 四、已经闭合与尚未闭合的边界

### 已闭合

1. **读状态闭环**：G1 DDS 状态 → SDK → 计算机显示/事件。
2. **查询闭环**：计算机请求 → G1 官方服务 → 计算机 JSON 结果。
3. **可见响应闭环**：计算机请求 → G1 扬声器播报 → 播放状态回读 → 计算机 JSON 结果。
4. **受限信息理解闭环**：计算机 JSON → schema/允许列表 → 已验证 SDK API。

### 尚未闭合

1. `rt/audio_msg` 播放状态与发起 TTS 的严格请求关联（当前只有请求后的时序观测）。
2. 服药任务的显式任务状态、确认语义和可恢复进度合同。
3. 跨进程请求幂等或持久化审计（当前只保证单进程、有限窗口）。
4. ASR 输入 → 意图层 → 响应的完整语音交互。
5. 摄像头、视觉、任务规划、Dex3、关节控制、运动控制和急停恢复。

## 五、当前 SDK 风险与事实

| 项目 | 当前事实 | 对后续的影响 |
|---|---|---|
| SDK listener 生命周期 | Gate 5F 已验证一轮连续 runtime 的 `CloseChannel()` → adapter/observer 销毁 → factory 释放 | 长时间运行、断网恢复和多轮启停仍未验收 |
| 请求幂等 | `CommandRuntimeCore` 已接入实机 runtime；同 ID 播报实际返回 `replayed=true` | 仅限单进程、30 秒窗口；重启后不会保留 |
| 音频频率 | 实机 runtime 已接入成功 TTS 最小 3 秒间隔 | 不同 ID 的批量播报策略尚未形成任务级合同 |
| 音频服务版本 | 服务可调用，但未回读 `vui_service` / module 版本 | 新音频能力须逐接口验证 |
| 物理控制 | 未调用 LowCmd、HandCmd、模式切换、Dex3 接口 | 这些接口没有从当前闭环继承权限或验收结论 |

## 六、面向 SDK 负责人的建议流程

### Phase S1：任务状态与播放关联合同

目标：在不扩大硬件命令的前提下，为后续照护任务定义请求、任务状态、确认和播放事件的唯一 owner。

Gate 5F 已完成：

- SDK 初始化、observer 注册、调用、超时、停止及进程退出顺序；
- `request_id` 结果重放和 TTS 速率门禁与实际 SDK 调用的连接；
- 每次 SDK 调用的结构化审计结果。

停止条件：任务状态能区分“请求已接收、SDK 已成功、播放已观察、用户已确认”，且重放不会重复推进状态。

### Phase S2：播放关联与输入扩展决策

目标：评估 `rt/audio_msg` 是否存在可验证的播放请求关联字段；若没有，保留当前时序观测语义，不扩大为 ASR。

首批命令仅对应已有 intent：

```text
查询音量 → get_volume
播报连接成功 → speak / connection_ok
```

停止条件：关联字段有实机证据，或报告明确保留“请求后观察到”的时序语义；未知句子、混合句、超长文本和重复请求均被拒绝或重放，不触发额外 SDK 调用。

### Phase S3：SDK 音频输入与状态扩展

目标：只读接入官方 `rt/audio_msg` 的 ASR 消息，统一送入同一 intent owner。

停止条件：ASR 原始消息、解析结果、置信度/结束标志和最终 intent 具有可追溯关联；不引入任意动作。

### Phase S4：长时运行与网络恢复专项验证

目标：在需要后台常驻前，专门验证长时运行、网络短断恢复、重复启停与资源边界。

停止条件：连续启停、网络断开恢复和进程退出均无 callback 悬空、资源增长或假成功。

## 七、后续总体路线

```text
SDK 可靠性与有限命令
→ 计算机信息接入（文件/程序状态/传感器结果）
→ ASR 或有限中文输入
→ 任务状态机与进度反馈
→ 独立定义视觉、Dex3、运动与恢复闭环
```

当前最值得投入的工作是 **S1 合同设计**。Gate 5F 已完成 20/20 网络稳定性、查询、固定播报、同 ID 重放与并行状态监视；下一步不扩展硬件动作，而是定义任务状态和播放关联语义。

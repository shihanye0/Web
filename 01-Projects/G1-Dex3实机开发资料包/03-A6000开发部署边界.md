# A6000 开发部署边界

## 推荐结论

**把 A6000 建成项目的主开发与训练工作站，但不要把它当成唯一的机器人控制机。**

从项目代码、Conda 环境、AI 编程工具、数据版本、基础模型到训练和评估，都建议以 A6000 为主。机器人侧仍需要一个靠近机器人、安装官方 SDK 的控制机，专门负责 DDS、相机采集和硬件安全。

```text
本地电脑
  Obsidian、浏览、轻量编辑、VS Code Remote-SSH
        |
        v
A6000
  Git 工作副本、Codex/Claude Code、Isaac Lab、LeRobot/π0.5、训练、评估、推理服务
        |
        | 受认证、可观测的网络接口（非裸 DDS 低层控制）
        v
机器人侧控制机
  官方 unitree_sdk2、相机采集、28D adapter、安全网关、DDS、急停/watchdog
        |
        v
G1 + Dex3 实机
```

## 为什么不让 A6000 直接控制低层 DDS

- A6000 是共享算力与训练服务器，长训练、GPU 抢占、网络抖动和重启都可能造成控制不确定性。
- `LowCmd` 和 `HandCmd_` 是硬件动作入口，必须由能在断网或推理超时后立即进入安全状态的本地网关管理。
- 实机安全限位、模式切换、急停和状态监控应独立于 VLA 服务；模型失败时不能依赖模型自己恢复。

A6000 可以运行 π0.5 推理，但机器人侧网关必须为每个响应设置超时：超时、非法 28D、相机断流、状态过期或硬件错误时，停止增量动作并回到已验证的安全行为。

## A6000 上应建立的独立环境

| 环境 | 建议职责 | 不应混装 |
|---|---|---|
| `lerobot312` | π0.5 训练、离线评估、HTTP 推理 | Isaac Sim/Isaac Lab 的 Python 依赖 |
| `isaaclab` | 仿真、回放、sim-to-real 回归 | π0.5 训练依赖树 |
| `g1_hw_sdk` | 官方 SDK 只读订阅、消息解析、硬件网关的测试副本 | Isaac Lab 渲染与训练包 |

现有项目已经采用“Isaac Lab Python 3.10 <-> HTTP <-> π0.5 Python 3.12”的拆分，这个边界应保留。不要试图在一个 Conda 环境混装 Isaac Lab、官方 SDK 和完整 LeRobot。

## AI 编程工具如何放置

- 在 A6000 安装 Codex CLI/Claude Code；在服务器项目根目录启动，让它们直接读取当前代码、日志和环境。
- 项目级 `AGENTS.md`、训练文档、环境导出文件和经过审核的 skills 可以跟随 Git 同步。
- API key、登录缓存、`auth.json`、SSH 私钥、整个 `~/.codex` / `~/.claude` 不复制；在 A6000 单独登录并最小化配置。
- 本地通过 VS Code Remote-SSH 连接 A6000 编辑即可，无需维护一份随时漂移的完整本地代码副本。

## 停止条件

在 A6000 完成训练不等于允许实机执行。只有 [[01-官方接口与到机验收]] 的映射和安全测试完成、硬件网关独立验收后，才进入低速动作试验。

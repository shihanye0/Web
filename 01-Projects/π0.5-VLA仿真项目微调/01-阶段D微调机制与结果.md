# 阶段 D 微调机制与结果

阶段 D 的目的不是获得比赛模型，而是在当前 RTX 4070 12GB 上证明一条最小训练闭环：数据能进入模型，梯度能反传，权重能保存，新进程能重载并输出合法动作。

## 阶段 D 实际做了什么

```text
7 条 LeRobot episode / 7195 帧 / 双目图像 / 任务文本 / 28D state-action
  -> QUANTILES 归一化
  -> state 离散为 256 档并拼入语言 prompt
  -> 图像缩放到 224x224
  -> 冻结的 PaliGemma 提取视觉和语言条件
  -> 动作专家学习 50 步 action chunk 的 flow-matching 速度场
  -> MSE loss
  -> 反向传播、梯度裁剪、SGD 更新
  -> 保存完整模型、processor 和训练状态
```

训练输入不是一张图对应一个单步动作。`chunk_size=50` 表示每个样本监督未来 50 个动作时间步；28 维动作在模型内部补到 32 维，但 loss 最终截回真实 28 维。

## 哪些参数被更新

阶段 D 设置：

```text
train_expert_only=true
freeze_vision_encoder=true
use_peft=false
```

`train_expert_only=true` 会冻结整个 PaliGemma，包括视觉塔、视觉投影和语言模型。真正训练的是：

| 组件                  |           参数量 | 是否更新           | 作用                                   |
| ------------------- | ------------: | -------------- | ------------------------------------ |
| PaliGemma VLM       | 3,449,982,704 | 否              | 提供冻结的图像/语言条件表示                       |
| Gemma action expert |   691,256,320 | 是              | 根据条件和噪声动作预测动作速度场                     |
| `action_in_proj`    |        33,792 | 是              | 32 维噪声动作投影到 expert hidden size       |
| `action_out_proj`   |        32,800 | 是              | expert 输出投影回 32 维动作                  |
| `time_mlp_in/out`   |     2,099,200 | 是              | 编码 flow-matching 时间并调制 action expert |
| 合计                  | 4,143,404,816 | 更新 693,422,112 | 约 16.7% 参数参与梯度更新                     |

`freeze_vision_encoder=true` 在“是否可训练”上被 `train_expert_only=true` 覆盖，但仍明确保持视觉模块为 eval 状态。阶段 D 不是全参数微调，也不是 LoRA。

## loss 在学什么

π0.5 使用 flow matching。对真实动作 chunk `a`、随机高斯噪声 `e` 和随机时间 `t`：

```text
x_t = t * e + (1 - t) * a
目标速度 u_t = e - a
模型预测 v_t = model(image, task, state, x_t, t)
loss = MSE(u_t, v_t)
```

直观理解：训练时把真实动作逐渐混入噪声，要求模型学会每个时间点应沿哪个方向移动；推理时从噪声出发，用 10 个去噪步骤积分得到 50 步动作 chunk。

## 输入是如何处理的

- state/action 使用数据集 stats 做 `QUANTILES` 归一化。
- 归一化 state 被离散到 256 个桶，以 `Task: ..., State: ...; Action:` 形式进入 tokenizer。
- 两路 `480x640` 图像缩放并 padding 到 `224x224`，数值从 `[0,1]` 转为 `[-1,1]`。
- 动作是绝对动作：`use_relative_actions=false`。
- 配置中虽然列出了亮度、对比度、仿射等变换，但 `image_transforms.enable=false`，本次没有数据增强。

## 为什么改用 SGD

π0.5 默认 preset 是 AdamW。阶段 D 首先真实尝试了 AdamW，但在第 1 步创建 optimizer state 时 OOM：693M 可训练参数需要两组一阶/二阶矩状态，若按 FP32 估算，仅这两组状态就约 5.17GiB，还未计模型、梯度和激活。

为只验证工程链路，阶段 D 关闭 policy preset，使用无 momentum SGD。它几乎不保存逐参数 optimizer state，实际 `optimizer_state.safetensors` 只有 16 bytes；代价是训练行为不再等价于正式 AdamW。

## 20 步发生了什么

- `batch_size=1`，完成 20 次参数更新。
- 可训练参数 693,422,112，总参数 4,143,404,816。
- loss 在 `0.351-1.521` 之间波动，不能用第 1 步和第 20 步判断收敛。
- 梯度范数均有限；训练循环在 optimizer step 前执行 `grad_clip_norm=1.0`。
- 峰值显存约 10.37GB。
- 训练主体约 17 秒，保存完整 checkpoint 后总进度约 23 秒；模型加载另需约 80 秒。
- cosine scheduler 因总步数只有 20，把 warmup 从 1000 自动缩放为 0，把 decay 缩放为 20 步。

## 结果应如何解释

新 checkpoint 在独立进程中完整重载，21 个固定样本全部输出有限 28 维动作，固定随机种子重复差为 0。平均 MAE 为 `0.159767`，base 为 `0.159774`，几乎没有变化。

正确结论是“训练工程闭环成立”；错误结论包括：

- 20 步已经学会养老照护任务。
- loss 最后一项较低或较高就代表训练好坏。
- SGD checkpoint 可以与历史 AdamW checkpoint 公平比较。
- 训练集固定样本 MAE 可以代表泛化或任务成功率。
- 28 维输出可以按当前错误手部标签直接下发 Dex3。

## 手部标签问题是否要求重训

当前 PI05 的 state prompt 只包含按索引排列的归一化数值，不包含关节名称；`action_feature_names` 只有在 relative action 处理启用时才参与转换，而阶段 D 的 `use_relative_actions=false`。因此标签错位没有改变本次 loss 的列对应关系。

推荐保守主线：

1. 保留原始数值列顺序作为 dataset canonical order。
2. 把 `info.json` 名称修正为真实源列名称。
3. 在 G1/Dex3 adapter 中显式执行 dataset order 到 DDS order 的重排。
4. 用逐列测试验证映射，再重新评估旧 checkpoint。

只有当团队决定把训练数据本身重排成 DDS 顺序时，旧 checkpoint 的输入/输出列合同才会改变，此时必须重训或建立严格等价的输入输出置换层。

# 开发方法论：Brainstorming → BDD → TDD

> 三个 Skill 的定位、区别和协作方式。
> 更新日期：2026-06-03

---

## 一句话总结

| Skill | 核心问题 | 阶段 |
|-------|---------|------|
| `/brainstorming` | **"做什么？怎么做？"** | 模糊想法 → 设计方案 |
| `/bdd` | **"系统具体该怎么表现？"** | 设计方案 → 行为规格 |
| `/tdd` | **"怎么确保代码正确？"** | 行为规格 → 可靠代码 |

**串起来就是：** `brainstorming（想清楚）→ bdd（写清楚）→ tdd（做出来）`

---

## 1. Brainstorming — 想清楚做什么

### 定位

创意发散、方案探索。在动手之前，把模糊的想法变成可执行的设计方案。

### 触发时机

- "我想做一个 XX 功能"（还不确定怎么做）
- 新功能开发前
- 需要技术选型或架构决策时

### 流程

```
探索项目上下文 → 提问澄清 → 提出 2-3 个方案 → 用户选方案 → 输出设计文档
```

### 输出物

设计文档：`docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`

包含：方案对比、权衡分析、推荐方案、技术选型。

### 关键约束

**有硬门禁（HARD-GATE）**：设计方案未获批准前，禁止写任何代码。

## 2. BDD — 写清楚系统行为

### 定位

行为驱动开发。用业务语言描述系统"该怎么做"，让技术人员和非技术人员对齐预期。

### 触发时机

- 方案已确定，需要定义具体行为
- "这个功能怎么验收"
- "前后端对齐一下接口行为"
- "有哪些边界情况"
- 梳理需求、写用户故事

### 核心格式：Given-When-Then

```
Given [前置条件]     — 系统处于什么状态
When  [触发动作]     — 用户做了什么
Then  [预期结果]     — 系统应该怎样响应
```

### 场景示例

```gherkin
Feature: 用户登录

  Scenario: 使用正确凭据登录
    Given 用户已注册邮箱 "user@example.com"
    When  用户输入正确的邮箱和密码并点击登录
    Then  应跳转到首页
    And   应显示欢迎消息

  Scenario: 使用错误密码登录
    Given 用户已注册邮箱 "user@example.com"
    When  用户输入错误的密码并点击登录
    Then  应显示错误消息 "邮箱或密码错误"
    And   应停留在登录页
```

### 场景编写规范

1. **一个场景只测一个行为** — 失败时能精确定位
2. **用业务语言** — 不写 API 路径、数据库查询
3. **标题描述行为** — 不用编号（"测试用例 1"❌）
4. **覆盖边界和异常** — 空值、极值、并发、安全

### 输出物

- Feature 文件（`.feature`）
- 验收标准文档
- 用户故事（As a... I want... So that...）

### 与 TDD 的关系

BDD 定义行为，TDD 驱动实现。BDD 场景可以直接映射到 TDD 测试：

| BDD 层 | TDD 层 |
|--------|--------|
| Given | Arrange（前置条件） |
| When | Act（触发行为） |
| Then | Assert（断言结果） |

## 3. TDD — 做出来且保证正确

### 定位

测试驱动开发。先写测试，再写代码，确保每一行代码都有测试覆盖。

### 触发时机

- 所有新功能开发
- 所有 Bug 修复
- 重构（行为不变但代码要改）

### 核心流程：Red-Green-Refactor

```
RED   → 写一个会失败的测试
GREEN → 写最少的代码让测试通过
REFACTOR → 优化代码，保持测试通过
重复...
```

### 铁律

```
没有失败的测试，就不能写生产代码。
```

写了代码再补测试？删掉，重来。

### 验证清单

- [ ] 每个新函数都有测试
- [ ] 每个测试都先看到它失败
- [ ] 每个测试因为正确原因失败（功能缺失，不是拼写错误）
- [ ] 写了最少代码让测试通过
- [ ] 所有测试通过
- [ ] 输出干净（无错误、无警告）

## 三者协作流程

```
模糊需求
   ↓
/brainstorming
   │  问清楚：做什么、约束是什么、成功标准是什么
   │  输出：2-3 个方案 + 推荐方案
   │  用户确认方案
   ↓
/bdd
   │  拆场景：正常流程、边界条件、异常、安全
   │  输出：Feature 文件 + 验收标准
   │  和产品/测试对齐
   ↓
/test-driven-development
   │  每个场景 → 写测试 → Red-Green-Refactor
   │  输出：通过测试的代码
   ↓
完成
```

### 实际案例：开发登录功能

**Step 1 — Brainstorming**
```
Boss: "我想做一个用户登录功能"

Claude: 用什么认证方式？JWT 还是 Session？
Boss: JWT

Claude: 需要支持第三方登录吗？
Boss: 先不要，后面再加

Claude: 要记住密码吗？
Boss: 要

→ 输出：设计方案（JWT + 记住密码 + 无第三方）
```

**Step 2 — BDD**
```
Feature: 用户登录

Scenario: 邮箱密码登录成功
Scenario: 密码错误提示错误
Scenario: 连续 5 次失败锁定账户
Scenario: 记住密码功能
Scenario: 未注册邮箱提示注册

→ 输出：5 个验收场景
```

**Step 3 — TDD**
```
为每个场景写测试：
test('rejects wrong password')  → RED
实现密码验证逻辑                → GREEN
重构：提取 validatePassword()   → REFACTOR

test('locks account after 5 failures') → RED
实现锁定逻辑                           → GREEN
...

→ 输出：通过所有测试的登录功能
```

## 什么时候用哪个

| 场景 | 用什么 |
|------|--------|
| "我想做个 XX"（不确定怎么做） | `/brainstorming` |
| "这个功能怎么验收"（已确定做什么） | `/bdd` |
| "帮我梳理需求"（可能两者都需要） | 先 `/brainstorming` 再 `/bdd` |
| "前后端对齐接口行为" | `/bdd` |
| "修个 bug" | 直接 `/tdd` |
| "纯重构，行为不变" | 直接 `/tdd` |
| "改配置" | 不需要任何 skill |

## 对比表

| 维度 | Brainstorming | BDD | TDD |
|------|--------------|-----|-----|
| **核心问题** | 做什么？怎么做？ | 系统该怎么表现？ | 代码对不对？ |
| **阶段** | 需求 → 设计 | 设计 → 行为规格 | 行为规格 → 代码 |
| **视角** | 产品/架构视角 | 业务/验收视角 | 开发/实现视角 |
| **语言** | 自然对话 | Given-When-Then | 编程语言测试 |
| **输出** | 设计文档 | Feature 文件 | 测试代码 |
| **参与者** | 开发 + 产品 | 开发 + 产品 + 测试 | 开发 |
| **硬约束** | 方案未批准不写代码 | 场景未对齐不写代码 | 没有失败测试不写代码 |
